# LogiTrack 設計仕様書

## 1. システム概要
残土運搬の流れ（船で積込港出港 → 荷下ろし港 → ダンプで最終埋立地へ搬入）を、
船ごとの残土が埋立完了するまで追跡する業務アプリ。Firebase 上で稼働。

## 2. 技術スタック
| 領域 | 採用 |
|---|---|
| 本体フロント | バニラ JavaScript（`app.js` / `index.html` / `style.css`） |
| データ | Firebase Firestore（compat SDK、project `unkou-final`） |
| ホスティング | Firebase Hosting（public = `hosting_main`） |
| OCR バックエンド | Cloud Functions v2（`functions/`、`extractSchedule`） |
| OCRエンジン | OpenAI gpt-4o（Vision）※キーは Functions の Secret `OPENAI_API_KEY` |
| 港スケジュール | Vite + React（`soma-port/`）→ ビルドして `/soma-port/` に同梱 |

## 5. KPI 算出（`renderKpis()` in app.js）
- 総積込量 = 進行中ロットのトン数合計 ＋ 履歴のトン数合計
- 船舶/ダンプ稼働数 = 稼働中（輸送中/積込待ち/積込中/荷卸し中）÷ 登録台数
- 最終埋立完了 = 荷卸し完了/完了ロット ＋ 履歴 のトン数合計
- 1ロットの代表トン数 = 重量t → 総t → 数量m³ → 積載 の優先順

## 6. ダンプ自動ステータス遷移
共通ヘルパー `getDumpTransportDate()` / `computeDumpPhase()`（app.js）で判定。
- 輸送日 = `dispatchDate`。未入力なら「登録日の翌日」。
- 輸送日 **07:00** 到達 → `輸送中`、**19:00** 到達 → 履歴（completed）。
- すべて**現地時刻（JST）のタイムスタンプ比較**。アプリ起動時に正しい段階へ追いつく（catch-up）。
- 状態変更は `checkAndUpdateStatuses()`（60秒間隔）に一元化。描画 `renderTruckTracking()` は表示専用。
- 過去に「2系統が併存」「`toISOString()` のUTC日付で朝の遷移がずれる」不具合があり統一済み。

## 7. 港入港スケジュール（soma-port）
- 別アプリ（Vite/React）。本体には `<iframe src="soma-port/">` で埋め込み。
  - ビルド済み `soma-port/` があれば相対参照、無ければ `localhost:5173`（Vite開発）にフォールバック。
- データは Firestore `artifacts/{appId}/public/data/schedules`（匿名ログイン）。
- 変更履歴は `artifacts/{appId}/public/data/scheduleLogs`（型 `MoveLog`）。

### OCRパイプライン
1. フロントが PDF を **縦3バンド（上・中・下）に分割**して画像化（各バンドに左端のバース名が必ず写る）。
2. `/api/extract-schedule` 経由で `extractSchedule` 関数へ POST。
3. 関数が gpt-4o に画像＋プロンプト（相馬港バース一覧・全行走査・重複排除）を渡し、JSON配列で受け取る。
4. 正規化して Firestore に洗い替え保存 → カレンダー表示。
- キー不正対策として関数側で `OPENAI_API_KEY.trim()`（末尾改行/空白の混入を除去）。

## 8. デプロイ構成
`firebase.json` の hosting ターゲット `unkou-final` → public `hosting_main`。
`deploy.ps1`:
1. `soma-port` をビルド（→ `hosting_public/soma-port`）
2. 正本（index.html/app.js/style.css/データ）を `hosting_main` へ同期
3. ビルド済み soma-port を `hosting_main/soma-port` へ配置
4. `firebase deploy --project unkou-final --only hosting:unkou-final`（`-WithFunctions` で関数も）

## 9. 既知の注意点 / 今後の候補
- OCRは高密度な予定表だと完璧にはならない → 手編集/D&Dで補正する運用前提。
- 自動遷移はブラウザ起動中に評価（常駐処理ではない）。完全自動化するなら Cloud Scheduler + Functions のサーバ側遷移が候補。
- 今後: ETA/ETD のドラッグ調整、日次画面でのバース間ドラッグ移動、soma-port のチャンク分割（pdf.worker が大きい）。
